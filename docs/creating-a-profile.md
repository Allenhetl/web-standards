# Creating a new profile

A **profile** is a site _type_. Adding one is **config, not new logic** — you
add a directory under `profiles/` and never touch `core/` or the reusable
workflows. Reach for this when a new site fits neither `jekyll-public` nor
`node-private` (e.g. a public Hugo site, a private Jekyll site, a Next.js
export).

## What a profile contains

```
profiles/<name>/
├── profile.yml            # the manifest (values, not logic)
├── README.md              # one paragraph: who it's for
├── caller-workflows/      # the CI stubs a site of this type installs
├── root-files/            # (only if headers/robots are STATIC & synced)
│   ├── _headers
│   └── robots.txt
└── headers-requirements.yml  # (only if headers are GENERATED & asserted)
```

Use **either** `root-files/_headers`+`robots.txt` (static, synced,
drift-checked) **or** `headers-requirements.yml` (generated by the site's
build, asserted) — matching how that site type manages headers. That single
choice is the main design decision.

## `profile.yml` keys

```yaml
name: <name> # must equal the directory name
runtime: <jekyll|node> # which build path build.yml / the CI workflows take
build-cmd: "<command>" # e.g. "bundle exec jekyll build" or "npm run build"
site-dir: "<dir>" # build output directory (e.g. _site or site)
codeql-languages: "<json-array or empty>" # "" = no CodeQL
headers: <static|generated> # static → synced from root-files/; generated → asserted
robots: <static|generated>
# node-only, optional:
validate-cmd: "<command>" # extra render gate; default off in CI (needs a browser)
```

`sync-standards.sh` / `check-drift.sh` already special-case
`headers: generated` (they skip syncing `_headers`/`robots.txt` for that
profile). No script change is needed for a new profile that reuses an
existing header strategy.

## Steps

1. **Branch** `web-standards`: `git checkout -b profile/<name>`.
2. **Copy the closest existing profile** and edit:
   ```bash
   cp -r profiles/jekyll-public profiles/<name>   # or node-private
   ```
   Edit `profile.yml` (name, runtime, build-cmd, site-dir, codeql-languages,
   headers/robots strategy) and `README.md`.
3. **Headers strategy:**
   - _Static_ → put the site type's `_headers` + `robots.txt` in
     `profiles/<name>/root-files/` (they'll be synced + drift-checked). Make
     sure the CSP meets `core/baseline-requirements.yml`.
   - _Generated_ → delete `root-files/`, add `headers-requirements.yml`
     listing the CSP directives + headers the build's output must contain
     (start from the site's _current_ generated output so it can only
     tighten). CI asserts it via `headers-assert.yml`.
4. **Caller workflows:** in `profiles/<name>/caller-workflows/`, keep the
   stubs this type needs and pass the right inputs. Reuse the existing
   reusable workflows — pass `runtime`/`build-cmd`/`site-dir` to
   `accessibility`/`lighthouse`/`broken-links`/`build`, `languages` to
   `codeql` (omit the stub entirely if `codeql-languages` is empty), and use
   `build-validate.yml` + `headers-assert.yml` for generated-header types.
   Set trigger branches to match the site (`main` and/or `master`).
5. **Runtime not yet supported by `build.yml`?** If the new type needs a
   third runtime (e.g. `hugo`, `python`), add a gated branch to
   `.github/workflows/build.yml` (and the a11y/lighthouse/broken-links
   workflows) behind `if: inputs.runtime == '<x>'`. This is the one case that
   touches shared workflows — keep every stack-specific step gated so other
   profiles are unaffected.
6. **Validate + document:** validate all YAML
   (`ruby -ryaml -e "YAML.load_file(ARGV[0])"` per file), add the profile to
   `README.md`'s profiles table and `docs/BUILDING-A-SITE.md`, and add a
   `CHANGELOG.md` entry.
7. **Release:** open a PR, merge, then bump the moving tag
   (`git tag -f v2 vX.Y.Z && git push -f origin v2`) — a new profile is
   additive, so it's a minor bump, not a new major.

## Rules of thumb

- **Never loosen the floor.** A profile may be stricter than
  `core/baseline-requirements.yml`, never looser — the assertion enforces
  this.
- **Don't copy logic.** If you're pasting build steps into a caller stub,
  they belong in a reusable workflow instead.
- **Prefer generated+asserted for anything sensitive.** Byte-drift is fine
  for a static public `_headers`; assertion is better when a build owns the
  output.
- **A profile is small.** If yours is growing large, the difference probably
  belongs in a parameter on a shared workflow, not in the profile.
